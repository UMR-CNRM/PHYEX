6 !
7 !IMPORTANT NOTE ON HOW TO DEAL WITH KEYS USED IN THE PHYSICS *AND* IN THE HOST MODEL
8 !
9 !When this situation occurs:
10 !For example, if a physics option (eg logical switch) has an impact on the model setup, this
11 !option must be accessible in the host model *and* in the physics.
12 !
13 !The prefered solution (when it is possible):
14 !The option is implemented in a module and in a namelist of PHYEX. And, after the initialisation
15 !is done, the option is available in the PHYEX_OUT structure (in addition to the PHYEX module) for use
16 !by the host model.
17 !
18 !When the physics option is needed in the host model before the PHYEX initialisation, or if the physics
19 !option must be computed from other options known by the host model, there are several solutions:
20 !
21 ! If this situation is specific to only one host model:
22 ! The option is declared (module) and initialised (namelist) twice: once in the host model, once in PHYEX;
23 ! and a consistency check is added after the physics initialisation.
24 ! This solution is preferred to overwriting the variable stored in the PHYEX module, because between
25 ! initialization and overwriting, the variable may have been used to initialize a parameterization.
26 !
27 ! If this issue is common to all the host models:
28 ! The option is declared and initialised in the host model. Then, the value is given, through INI_PHYEX, to
29 ! the parametrisation INIT subroutine (eg TURBN_INIT). In the INIT subroutine the value is assigned
30 ! to a module variable specific to PHYEX (in order to be accessible from inside the parametrisations).
31 ! In this case the variable *must not* be added in a PHYEX namelist.
32 ! An alternative to this solution is not to copy the variable inside a PHYEX module but add it directly
33 ! to the parametrisation call arguments.
34 !
